## Domain Question: DD Items and Deal Stages

**The relationship between DD items and deal stages is temporal, not enforced.** The schema places no foreign-key constraint between `DueDiligenceItem` and `DealStage` — a DD item can be created on any deal at any stage. However, DD items only carry business meaning when the deal has progressed far enough that formal diligence is underway.

**The DD-active stages are:**

| Stage | Why DD is relevant |
|-------|-------------------|
| `DUE_DILIGENCE` | Primary DD phase. The deal has passed LOI and the fund is formally investigating the target. Most DD items are created and worked here. |
| `FINAL_NEGOTIATION` | Diligence continues. Open items (especially legal, tax, environmental) frequently remain unresolved as terms are negotiated. A GP checking DD status during final negotiation is looking for "are there any blockers before we close?" |
| `CLOSING` | Last-mile diligence. Conditions precedent (insurance, regulatory clearance, title transfer) may still be outstanding. DD items in this stage are high-stakes — a red flag here can kill a deal. |

**Stages explicitly excluded from the DD overview:**

| Stage | Reason |
|-------|--------|
| `IDENTIFIED` through `LOI_NEGOTIATION` | Pre-DD. While a fund manager *can* create DD items early (e.g., preliminary legal review during NDA phase), these deals should not appear in the DD overview because the deal hasn't committed to formal diligence. If a user wants DD items for an early-stage deal, they use `getDealDDItems` with the deal name — the detail tool has no stage filter. |
| `CLOSED_WON` | Diligence is complete. DD items are historical record, not action items. |
| `CLOSED_LOST`, `CLOSED`, `PASSED`, `ON_HOLD` | Terminal or paused. Not actionable. |

**The stage filtering logic:**

```
DD_ACTIVE_STAGES = ['DUE_DILIGENCE', 'FINAL_NEGOTIATION', 'CLOSING']
```

- **Overview tool (`getDDOverview`)**: Queries deals in `DD_ACTIVE_STAGES` that have at least one DD item. This answers "what's my active diligence workload across the fund?"
- **Detail tool (`getDealDDItems`)**: No stage filter. If the user asks about a specific deal's DD, show everything regardless of stage. The user already knows which deal they want — the tool shouldn't gate-keep.

This is the same pattern as capital calls: the summary tool scopes to "active" records, while the detail tool returns whatever the user asks for.

## Tool 1: getDDOverview

### 1. JTBD

> "The user needs to see the diligence workload across all active deals at a glance — how many items are open, which deals have red flags, and where progress is stalled — so they can allocate analyst time and identify deals at risk of timeline slippage before it impacts closing."

### 2. Kano Classification

**Basic** — A fund manager running 3-5 deals through DD simultaneously expects to ask "how's diligence going?" and get a cross-deal answer. Without this, they must open each deal individually in the dashboard. This is the DD equivalent of `getCapitalCallSummary` — the fund-level overview that establishes context before drilling down.

### 3. Data Availability

```
Models: Deal (parent, stage filter), DueDiligenceItem (DD data)

DueDiligenceItem fields used:
  - dealId         → join to Deal
  - category       → DDCategory enum (16 values)
  - status         → DDStatus enum (6 values)
  - redFlag        → boolean
  - priority       → integer (1-5)

Deal fields used:
  - id, name, companyName, stage, fundId
  - deletedAt      → soft-delete filter required on Deal

Soft delete: DueDiligenceItem has NO deletedAt (hard-cascades from Deal).
             Deal has deletedAt → apply ...notDeleted.

Fund isolation: DueDiligenceItem.dealId → Deal.fundId (indirect).
```

All required fields exist in the schema. No new columns or migrations needed.

### 3. Tool Signature

```
name:        getDDOverview
description: Get the due diligence status across all active deals in the fund.
             Shows completion progress, red flags, and category breakdown for
             deals in DUE_DILIGENCE, FINAL_NEGOTIATION, and CLOSING stages.
category:    deals
input:       {} (no parameters)
```

**Output shape:**
- `totalDeals` — count of deals in DD-active stages with DD items
- `totalItems` — total DD items across all active deals
- `completedItems` — count with status COMPLETED or NA
- `overallProgress` — percentage (completedItems / totalItems)
- `totalRedFlags` — count of items with redFlag = true
- `deals` — array of per-deal summaries:
  - `dealId`, `dealName`, `companyName`, `stage`
  - `totalItems`, `completedItems`, `progress` — percentage
  - `redFlagCount` — integer
  - `pendingHighPriority` — count of items with priority <= 2 and status not COMPLETED/NA
  - `byCategory` — array of `{ category, total, completed, redFlags }`

### 4. Prisma Query Sketch

```
Models: Deal, DueDiligenceItem

Query — Deals in DD-active stages with their DD items:
  const DD_ACTIVE_STAGES = ['DUE_DILIGENCE', 'FINAL_NEGOTIATION', 'CLOSING']

  prisma.deal.findMany({
    where: {
      fundId: ctx.fundId,
      ...notDeleted,
      stage: { in: DD_ACTIVE_STAGES },
      dueDiligenceItems: { some: {} },   // only deals with at least one DD item
    },
    include: {
      dueDiligenceItems: {
        select: {
          category: true,
          status: true,
          redFlag: true,
          priority: true,
        },
      },
    },
    orderBy: { name: 'asc' },
  })

Aggregation: computed in JS per deal and across all deals.
Cardinality is low — a fund typically has 3-8 deals in DD stages,
each with 20-80 DD items. Total items < 500 per fund.
```

### 5. Empty State

```json
{
  "totalDeals": 0,
  "totalItems": 0,
  "completedItems": 0,
  "overallProgress": "0%",
  "totalRedFlags": 0,
  "deals": [],
  "summary": "No deals are currently in due diligence. Deals enter the DD phase after LOI acceptance — DD items are created to track investigation across financial, legal, operational, and other categories."
}
```

### 6. Currency Handling

No monetary fields in this tool. DD items are tracking items, not financial records. No `formatMoney` calls needed.

## Tool 2: getDealDDItems

### 1. JTBD

> "The user needs to drill into a specific deal's due diligence to see every item — grouped by category, with status, priority, findings, and red flags — so they can assess whether the deal is ready to advance to the next stage or whether open items need resolution first."

### 2. Kano Classification

**Performance** — The overview answers "which deals need attention?" This tool answers "what specifically is open on Acme Corp?" The deeper the drill, the more Emma replaces the spreadsheet-based DD tracker that most search funds use today. A GP asking "are there any red flags on Acme?" expects an instant, complete answer.

### 3. Data Availability

```
Models: Deal (parent, name/ID lookup), DueDiligenceItem (full detail)

DueDiligenceItem fields used:
  - ALL fields (id, category, item, status, priority, assignedTo,
    notes, findings, redFlag, createdAt, updatedAt)

Deal fields used:
  - id, name, companyName, stage, fundId
  - deletedAt → soft-delete filter on Deal

No stage filter on this tool — if the user asks about a specific
deal's DD, show everything.

Soft delete: DueDiligenceItem has NO deletedAt. Deal has deletedAt.
```

All required fields exist. No new columns or migrations needed.

### 3. Tool Signature

```
name:        getDealDDItems
description: Get all due diligence items for a specific deal, grouped by category.
             Includes status, priority, findings, and red flags for each item.
             Use this when the user asks about DD status for a particular deal.
category:    deals
input:       {
               dealNameOrId: string  // Deal name (partial match) or deal ID.
                                     // Uses the same lookup pattern as getDealDetails.
             }
```

**Output shape:**
- `dealName`, `companyName`, `stage`
- `totalItems`, `completedItems`, `progress` — percentage
- `redFlagCount` — integer
- `byCategory` — array of category groups:
  - `category` — DDCategory enum value
  - `total`, `completed`, `redFlags`
  - `items` — array of DD items in that category:
    - `id`, `item` (description), `status`, `priority`
    - `assignedTo` — string or null
    - `findings` — string or null (investigation results)
    - `redFlag` — boolean
    - `updatedAt` — ISO date

### 4. Prisma Query Sketch

```
Models: Deal, DueDiligenceItem

Query — Deal by name/ID with all DD items:
  const deal = await prisma.deal.findFirst({
    where: {
      fundId: ctx.fundId,
      ...notDeleted,
      OR: [
        { id: dealNameOrId },
        { name: { contains: dealNameOrId, mode: 'insensitive' } },
      ],
    },
    include: {
      dueDiligenceItems: {
        orderBy: [
          { category: 'asc' },
          { priority: 'asc' },
          { createdAt: 'asc' },
        ],
      },
    },
  })

Same name/ID lookup pattern as getDealDetails — try ID first via
the OR clause, fall back to partial name match.

Post-processing: group items by category in JS, compute per-category
and overall aggregates.
```

### 5. Empty State

**Deal not found:**

```json
{
  "error": "Deal not found. Check the deal name or use getPipelineSummary to see all active deals."
}
```

**Deal found but no DD items:**

```json
{
  "dealName": "Acme Corp",
  "companyName": "Acme Corporation",
  "stage": "DUE_DILIGENCE",
  "totalItems": 0,
  "completedItems": 0,
  "progress": "0%",
  "redFlagCount": 0,
  "byCategory": [],
  "summary": "No due diligence items have been created for this deal yet. DD items are typically organized by category (Financial, Legal, Commercial, etc.) and track the investigation of each diligence area."
}
```

### 6. Currency Handling

No monetary fields. DD items track investigation status, not financial values.

## Tool 3: getDealContacts

### 1. JTBD

> "The user needs to quickly pull up the contacts associated with a deal — who's the owner, who's the broker, who's the attorney — so they can coordinate outreach, schedule meetings, or find a specific person's email without leaving the conversation."

### 2. Kano Classification

**Basic** — A GP saying "who's the broker on the Acme deal?" or "give me the owner's email" is a frequent, high-friction question today. The alternative is navigating to the deal page and scanning the contacts list. This is a table-stakes lookup that makes Emma usable as a quick-reference operating partner during calls or meetings.

### 3. Data Availability

```
Models: Deal (parent, name/ID lookup), DealContact (contact data)

DealContact fields used:
  - ALL fields (id, name, title, email, phone, role, isPrimary, notes)

DealContactRole enum: OWNER, CO_OWNER, MANAGEMENT, BROKER, ATTORNEY,
                      ACCOUNTANT, ADVISOR, OTHER

Deal fields used:
  - id, name, companyName, fundId
  - deletedAt → soft-delete filter on Deal

Soft delete: DealContact has NO deletedAt (hard-cascades from Deal).
             Deal has deletedAt → apply ...notDeleted.
```

All required fields exist. No new columns or migrations needed.

### 3. Tool Signature

```
name:        getDealContacts
description: Get all contacts associated with a specific deal including names,
             roles, titles, and contact information. Use this when the user asks
             about who is involved in a deal or needs someone's contact details.
category:    deals
input:       {
               dealNameOrId: string  // Deal name (partial match) or deal ID.
             }
```

**Output shape:**
- `dealName`, `companyName`
- `totalContacts` — integer
- `primaryContact` — the contact with `isPrimary = true`, or null
- `contacts` — array sorted by role priority:
  - `name`, `title` (or null), `role` — DealContactRole enum
  - `email` (or null), `phone` (or null)
  - `isPrimary` — boolean
  - `notes` — string or null

### 4. Prisma Query Sketch

```
Models: Deal, DealContact

Query — Deal by name/ID with contacts:
  const deal = await prisma.deal.findFirst({
    where: {
      fundId: ctx.fundId,
      ...notDeleted,
      OR: [
        { id: dealNameOrId },
        { name: { contains: dealNameOrId, mode: 'insensitive' } },
      ],
    },
    include: {
      contacts: {
        orderBy: [
          { isPrimary: 'desc' },   // Primary contact first
          { role: 'asc' },         // Then alphabetical by role
          { name: 'asc' },         // Then by name
        ],
      },
    },
  })

No soft-delete filter needed on DealContact itself — no deletedAt field.
Fund isolation through Deal.fundId + notDeleted on Deal.
```

### 5. Empty State

**Deal not found:**

```json
{
  "error": "Deal not found. Check the deal name or use getPipelineSummary to see all active deals."
}
```

**Deal found but no contacts:**

```json
{
  "dealName": "Acme Corp",
  "companyName": "Acme Corporation",
  "totalContacts": 0,
  "primaryContact": null,
  "contacts": [],
  "summary": "No contacts have been added to this deal yet. Contacts typically include the business owner, broker, attorney, and key management team members."
}
```

### 6. Currency Handling

No monetary fields. Contact data is non-financial.

## Cross-Cutting Decisions

### Tool Count Impact on Prompt

Adding 3 tools brings the total from 10 to 13. The prompt `[AVAILABLE TOOLS]` section grows by approximately 700 characters (~175 tokens). At the current 50K token budget, this is 0.35% of context — negligible. The `ToolRegistry.generatePromptSection()` handles this automatically.

### File Structure

```
src/lib/ai/tools/deals/
├── get-pipeline-summary.ts    (existing)
├── get-deal-details.ts        (existing)
├── get-dd-overview.ts         (new)
├── get-deal-dd-items.ts       (new)
└── get-deal-contacts.ts       (new)
```

### Registration

Each tool registered in `create-default-registry.ts` — one import + one `registry.register()` call per tool.

```typescript
// Phase 1.5 Batch 2: Due diligence & contact tools
registry.register(getDDOverview)
registry.register(getDealDDItems)
registry.register(getDealContacts)
```

### Soft Delete Awareness

- `Deal` has `deletedAt`. Apply `...notDeleted` on all Deal queries.
- `DueDiligenceItem` has NO `deletedAt` — cascades from Deal. No soft-delete filter needed on DD items.
- `DealContact` has NO `deletedAt` — cascades from Deal. No soft-delete filter needed on contacts.

This is the same cascade pattern as `CapitalCallItem` and `DistributionItem` in Batch 1.

### Stage Filtering Constants

```typescript
const DD_ACTIVE_STAGES: DealStage[] = [
  'DUE_DILIGENCE',
  'FINAL_NEGOTIATION',
  'CLOSING',
]
```

This constant is used ONLY in `getDDOverview`. The detail tools (`getDealDDItems`, `getDealContacts`) have no stage filter because the user is requesting a specific deal by name — stage is informational, not a gate.

### Existing Tool Overlap

`getDealDetails` returns deal financials (asking price, revenue, EBITDA, multiples) but surfaces **zero DD items and zero contacts**. Batch 2 tools are strictly additive:

| Question | Before Batch 2 | After Batch 2 |
|----------|----------------|---------------|
| "How's diligence going?" | Manual dashboard check | `getDDOverview` |
| "What's open on Acme's DD?" | Not possible via Emma | `getDealDDItems` |
| "Who's the broker on Acme?" | Not possible via Emma | `getDealContacts` |
| "Tell me about Acme" | `getDealDetails` (financials) | Same — no change |

### Test Strategy

Each tool gets entries in `tool-parity.test.ts`:
- Name, description, and category validated against `EXPECTED_TOOL_METADATA`
- SDK shape verified (description, inputSchema, execute)
- `generatePromptSection` includes all tool names

The `prompt-snapshot.test.ts` baseline (`OLD_APPROX_TOKEN_COUNT`) will need updating after adding 3 tools — same calibration pattern as Batch 1.
