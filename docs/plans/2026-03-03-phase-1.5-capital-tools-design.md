## Tool 1: getCapitalCallSummary

### 1. JTBD

> "The user needs to see all capital calls at a glance — how many are outstanding, how much has been called vs. collected, and whether the fund's capital call schedule is on track — so they can assess LP funding health without opening every call individually."

### 2. Kano Classification

**Basic** — A fund manager with active capital calls expects to ask "what's my capital call status?" and get an immediate, complete answer. Absence of this tool forces them to leave Emma and check the dashboard manually, breaking the conversational flow.

### 3. Tool Signature

```
name:        getCapitalCallSummary
description: Get a summary of all capital calls for the fund including total called,
             collected, outstanding amounts, and status breakdown by call.
category:    capital
input:       {} (no parameters)
```

**Output shape:**
- `totalCalls` — count of all non-draft, non-cancelled calls
- `totalCalled` — sum of all call totalAmounts (formatted currency)
- `totalCollected` — sum of all item paidAmounts across all calls (formatted currency)
- `totalOutstanding` — totalCalled minus totalCollected (formatted currency)
- `collectionRate` — totalCollected / totalCalled as percentage
- `byStatus` — array of `{ status, count, amount }` grouped by CapitalCallStatus
- `calls` — array of recent calls (last 10), each with:
  - `callNumber`, `callDate`, `dueDate`, `totalAmount`, `status`, `purpose`
  - `itemsPaid` / `itemsTotal` — e.g., "8/10 LPs funded"
  - `collectedAmount` — sum of item paidAmounts for that call

### 4. Prisma Query Sketch

```
Models: CapitalCall, CapitalCallItem

Query 1 — All active calls:
  prisma.capitalCall.findMany({
    where: {
      fundId: ctx.fundId,
      status: { notIn: ['DRAFT', 'CANCELLED'] },
    },
    include: {
      items: {
        select: { callAmount, paidAmount, status },
      },
    },
    orderBy: { callNumber: 'desc' },
  })

Aggregation: computed in JS from the included items.
No separate aggregate query needed — capital calls per fund are low cardinality
(typically 2-20 calls total across a fund's lifetime).
```

### 5. Empty State

```json
{
  "totalCalls": 0,
  "totalCalled": "$0",
  "totalCollected": "$0",
  "totalOutstanding": "$0",
  "collectionRate": "0%",
  "byStatus": [],
  "calls": [],
  "summary": "No capital calls have been issued for this fund. Capital calls are created when the fund needs to draw down committed capital from LPs — typically for acquisitions, fees, or expenses."
}
```

The `summary` field provides context so Emma can educate the user rather than returning a silent empty array.

### 6. Currency Handling

All monetary fields (`totalCalled`, `totalCollected`, `totalOutstanding`, per-call `totalAmount`, `collectedAmount`) pass through `formatMoney(value, ctx.currency)`. No hardcoded symbols. The `collectionRate` uses `formatPercent()`.

## Tool 2: getCapitalCallDetails

### 1. JTBD

> "The user needs to drill into a specific capital call to see exactly which LPs have paid, which haven't, and how much is outstanding per investor — so they can follow up with delinquent LPs or confirm funding before closing a deal."

### 2. Kano Classification

**Performance** — The summary tool answers "how are capital calls going overall?" This tool answers "what's happening with Call #3 specifically?" The deeper the fund manager can drill without leaving the conversation, the more valuable Emma becomes as an operating partner.

### 3. Tool Signature

```
name:        getCapitalCallDetails
description: Get detailed information about a specific capital call including
             per-investor payment status, amounts, and collection progress.
category:    capital
input:       {
               callNumberOrId: string  // Call number (e.g., "3") or call ID.
                                       // Numeric strings are matched against
                                       // callNumber first, then ID.
             }
```

**Output shape:**
- `callNumber`, `status`, `callDate`, `dueDate`, `noticeDate`, `completedDate`
- `totalAmount`, `forInvestment`, `forFees`, `forExpenses` — all formatted currency
- `purpose`, `dealReference`
- `collectedAmount`, `outstandingAmount` — computed from items
- `collectionRate` — percentage
- `isOverdue` — boolean (dueDate < now AND status not FULLY_FUNDED/CANCELLED)
- `daysOverdue` — integer (0 if not overdue)
- `items` — array of per-LP line items:
  - `investorName`, `investorType`
  - `callAmount`, `paidAmount`, `outstandingAmount` — formatted currency
  - `status` (PENDING | NOTIFIED | PARTIAL | PAID | OVERDUE | DEFAULTED)
  - `paidDate` (ISO date or null)

### 4. Prisma Query Sketch

```
Models: CapitalCall, CapitalCallItem, Investor

Query — Single call with items and investor names:
  prisma.capitalCall.findFirst({
    where: {
      fundId: ctx.fundId,
      OR: [
        { callNumber: parseInt(callNumberOrId) || -1 },
        { id: callNumberOrId },
      ],
    },
    include: {
      items: {
        include: {
          investor: {
            select: { name: true, type: true },
          },
        },
        orderBy: { callAmount: 'desc' },  // Largest commitments first
      },
    },
  })

Note: OR clause handles both "3" (matched as callNumber) and cuid (matched as id).
parseInt fallback to -1 prevents matching callNumber 0.
```

### 5. Empty State

```json
{
  "error": "Capital call not found. Check the call number or use getCapitalCallSummary to see all active calls."
}
```

This is a lookup tool — the empty state is "not found" with a redirect to the summary tool, not a generic error.

### 6. Currency Handling

All monetary fields (`totalAmount`, `forInvestment`, `forFees`, `forExpenses`, per-item `callAmount`, `paidAmount`, `outstandingAmount`) pass through `formatMoney(value, ctx.currency)`. `collectionRate` uses `formatPercent()`.

## Tool 3: getOverdueCallItems

### 1. JTBD

> "The user needs to immediately see which specific LPs have missed their capital call payments and by how much — so they can prioritize follow-up calls, assess default risk, and decide whether to issue cure notices before the situation escalates."

### 2. Kano Classification

**Basic** — This is the highest-anxiety data point for a GP. If an LP defaults on a capital call, it can jeopardize a closing. A $500M fund GP checking this weekly would immediately notice if the logic was imprecise. Getting this wrong is worse than not having it.

### 3. Overdue Definition (Domain Precision)

**An item is overdue when it meets ALL of the following:**

1. The parent `CapitalCall.status` is `SENT` or `PARTIALLY_FUNDED` (the call is active and has been issued to LPs — DRAFT and APPROVED calls haven't been sent yet, so LPs can't be delinquent on them)
2. The parent `CapitalCall.dueDate < today` (the payment deadline has passed)
3. The `CapitalCallItem.status` is NOT `PAID` (the LP hasn't fully funded their share)

**This intentionally uses a dual-signal approach:**

- **Date-based:** `dueDate < now()` catches items even if the system hasn't been updated to mark them as `OVERDUE` in the status field. This is critical because status updates depend on a human or batch process — the date never lies.
- **Status-inclusive:** Items already marked `OVERDUE` or `DEFAULTED` in `CallItemStatus` are always included regardless of date math, because a GP may have manually flagged them.
- **Exclusion logic:** Items with status `PAID` are excluded even if the call is overdue. Items with status `PARTIAL` are included — a partial payment still means outstanding capital.

**The combined predicate:**

```
(call.status IN ['SENT', 'PARTIALLY_FUNDED'])
AND (call.dueDate < now())
AND (item.status NOT IN ['PAID'])
```

OR independently:

```
(item.status IN ['OVERDUE', 'DEFAULTED'])
AND (call.status NOT IN ['DRAFT', 'CANCELLED', 'FULLY_FUNDED'])
```

**Granularity: per-LP items, not per-call.** The tool name is `getOverdueCallItems` — it returns individual LP line items, grouped by their parent call for context. If Call #3 has 10 LPs and 8 have paid, the tool surfaces the 2 delinquent LPs with their specific amounts, not the entire call marked as overdue.

**Partially funded calls:** A call with status `PARTIALLY_FUNDED` is the primary source of overdue items. The tool filters to the unpaid items within that call. A GP seeing "Call #3: 2 of 10 LPs outstanding, $450,000 uncollected" gets precisely the information they need for follow-up.

### 3. Tool Signature

```
name:        getOverdueCallItems
description: Get all overdue capital call line items — specific LPs who have
             missed payment deadlines. Returns per-investor detail grouped by
             call, including amounts owed, days overdue, and contact info for
             follow-up. Only includes calls that have been sent to LPs.
category:    capital
input:       {} (no parameters)
```

**Output shape:**
- `totalOverdueItems` — count of delinquent LP items
- `totalOverdueAmount` — sum of (callAmount - paidAmount) for all overdue items (formatted currency)
- `overdueByCall` — array grouped by parent call:
  - `callNumber`, `dueDate`, `daysOverdue`, `totalAmount`, `purpose`
  - `items` — array of delinquent LP items:
    - `investorName`, `investorType`
    - `callAmount` — what the LP owes (formatted currency)
    - `paidAmount` — what the LP has paid so far (formatted currency)
    - `outstandingAmount` — callAmount minus paidAmount (formatted currency)
    - `status` (PENDING | NOTIFIED | PARTIAL | OVERDUE | DEFAULTED)
    - `contactName`, `contactEmail` — for GP follow-up (from Investor record)
    - `daysOverdue` — integer, computed from call dueDate

### 4. Prisma Query Sketch

```
Models: CapitalCall, CapitalCallItem, Investor

Query — Overdue items via dual-signal:
  const now = new Date()

  prisma.capitalCallItem.findMany({
    where: {
      status: { notIn: ['PAID'] },
      capitalCall: {
        fundId: ctx.fundId,
        OR: [
          // Signal 1: date-based (call sent + past due + item not paid)
          {
            status: { in: ['SENT', 'PARTIALLY_FUNDED'] },
            dueDate: { lt: now },
          },
          // Signal 2: status-based (item explicitly flagged)
          // Handled by outer status filter — items marked OVERDUE/DEFAULTED
          // on non-terminal calls are always captured
        ],
        status: { notIn: ['DRAFT', 'CANCELLED', 'FULLY_FUNDED'] },
      },
    },
    include: {
      capitalCall: {
        select: {
          callNumber: true,
          dueDate: true,
          totalAmount: true,
          purpose: true,
          status: true,
        },
      },
      investor: {
        select: {
          name: true,
          type: true,
          contactName: true,
          contactEmail: true,
        },
      },
    },
    orderBy: {
      capitalCall: { dueDate: 'asc' },  // Oldest overdue first
    },
  })

Post-processing: group items by capitalCall.id in JS.
Compute daysOverdue = Math.floor((now - call.dueDate) / 86400000).
```

### 5. Empty State

```json
{
  "totalOverdueItems": 0,
  "totalOverdueAmount": "$0",
  "overdueByCall": [],
  "summary": "No overdue capital call items. All LPs with outstanding calls have either paid in full or their calls are not yet past due."
}
```

This is the "green light" response. The GP's anxiety is addressed directly: no one is delinquent. The wording avoids ambiguity — "not yet past due" clarifies that there may be pending calls, they're just not late.

### 6. Currency Handling

All monetary fields (`totalOverdueAmount`, per-item `callAmount`, `paidAmount`, `outstandingAmount`, per-call `totalAmount`) pass through `formatMoney(value, ctx.currency)`. No hardcoded symbols.

## Tool 4: getDistributionSummary

### 1. JTBD

> "The user needs to see the full distribution history — how much has been returned to LPs, in what form (return of capital vs. profit vs. dividends), and the current distribution pipeline — so they can report to LPs on realized returns and plan upcoming distributions."

### 2. Kano Classification

**Basic** — LPs ask "when do I get my money back?" more than any other question. The GP needs distribution data at their fingertips to maintain LP confidence. This is table-stakes for a fund management platform.

### 3. Tool Signature

```
name:        getDistributionSummary
description: Get a summary of all distributions for the fund including total
             distributed by type (return of capital, profit, dividends),
             status breakdown, and recent distribution history.
category:    capital
input:       {} (no parameters)
```

**Output shape:**
- `totalDistributions` — count of all non-draft, non-cancelled distributions
- `totalDistributed` — sum of all distribution totalAmounts (formatted currency)
- `byType` — breakdown by DistributionType:
  - `{ type, count, amount }` for each of: RETURN_OF_CAPITAL, PROFIT_DISTRIBUTION, RECALLABLE, FINAL, SPECIAL
- `byStatus` — array of `{ status, count, amount }` grouped by DistributionStatus
- `totalReturnOfCapital` — formatted currency (sum of returnOfCapital across all distributions)
- `totalRealizedGains` — formatted currency
- `totalDividends` — formatted currency
- `totalInterest` — formatted currency
- `distributions` — array of recent distributions (last 10):
  - `distributionNumber`, `distributionDate`, `type`, `status`
  - `totalAmount` — formatted currency
  - `source` — what generated this (exit, dividend, etc.)
  - `itemsPaid` / `itemsTotal` — e.g., "9/10 LPs paid"

### 4. Prisma Query Sketch

```
Models: Distribution, DistributionItem

Query — All active distributions:
  prisma.distribution.findMany({
    where: {
      fundId: ctx.fundId,
      ...notDeleted,
      status: { notIn: ['DRAFT', 'CANCELLED'] },
    },
    include: {
      items: {
        select: { grossAmount, netAmount, status },
      },
    },
    orderBy: { distributionNumber: 'desc' },
  })

Aggregation: computed in JS from the results.
Same rationale as capital calls — low cardinality per fund (typically 1-15 distributions).
Breakdown by type/status grouped in JS from the flat list.
```

### 5. Empty State

```json
{
  "totalDistributions": 0,
  "totalDistributed": "$0",
  "byType": [],
  "byStatus": [],
  "totalReturnOfCapital": "$0",
  "totalRealizedGains": "$0",
  "totalDividends": "$0",
  "totalInterest": "$0",
  "distributions": [],
  "summary": "No distributions have been made from this fund. Distributions occur when the fund returns capital to LPs — typically from portfolio company exits, dividends, or recapitalization events."
}
```

### 6. Currency Handling

All monetary fields (`totalDistributed`, `totalReturnOfCapital`, `totalRealizedGains`, `totalDividends`, `totalInterest`, per-distribution `totalAmount`) pass through `formatMoney(value, ctx.currency)`. No hardcoded symbols.

## Tool 5: getDistributionDetails

### 1. JTBD

> "The user needs to drill into a specific distribution to see the per-LP breakdown — gross amount, withholding tax, net amount, and payment status — so they can confirm that each investor received the correct amount and reconcile any payment issues."

### 2. Kano Classification

**Performance** — The summary answers "what's been distributed?" This tool answers "who got what from Distribution #2?" Essential for reconciliation, LP inquiries, and audit preparation. A GP fielding an LP's question about a specific distribution needs per-investor precision.

### 3. Tool Signature

```
name:        getDistributionDetails
description: Get detailed information about a specific distribution including
             per-investor payment amounts, withholding tax, net amounts, and
             payment status.
category:    capital
input:       {
               distributionNumberOrId: string  // Distribution number (e.g., "2")
                                               // or distribution ID. Numeric strings
                                               // match distributionNumber first.
             }
```

**Output shape:**
- `distributionNumber`, `type`, `status`, `distributionDate`, `approvedDate`, `paidDate`
- `totalAmount`, `returnOfCapital`, `realizedGains`, `dividends`, `interest` — all formatted currency
- `source`, `notes`
- `totalGross` — sum of item grossAmounts (formatted currency)
- `totalWithholding` — sum of item withholdingTax (formatted currency)
- `totalNet` — sum of item netAmounts (formatted currency)
- `items` — array of per-LP line items:
  - `investorName`, `investorType`
  - `grossAmount`, `withholdingTax`, `netAmount` — formatted currency
  - `status` (PENDING | PROCESSING | PAID | FAILED)
  - `paidDate` (ISO date or null)
  - `paymentMethod` (wire, ACH, check, etc. — or null)

### 4. Prisma Query Sketch

```
Models: Distribution, DistributionItem, Investor

Query — Single distribution with items and investor names:
  prisma.distribution.findFirst({
    where: {
      fundId: ctx.fundId,
      ...notDeleted,
      OR: [
        { distributionNumber: parseInt(distributionNumberOrId) || -1 },
        { id: distributionNumberOrId },
      ],
    },
    include: {
      items: {
        include: {
          investor: {
            select: { name: true, type: true },
          },
        },
        orderBy: { grossAmount: 'desc' },  // Largest allocations first
      },
    },
  })

Same OR-clause pattern as getCapitalCallDetails for number/ID disambiguation.
```

### 5. Empty State

```json
{
  "error": "Distribution not found. Check the distribution number or use getDistributionSummary to see all distributions."
}
```

Same pattern as getCapitalCallDetails — lookup tool returns "not found" with redirect to summary tool.

## Cross-Cutting Decisions

### Tool Count Impact on Prompt

Adding 5 tools brings the total from 5 to 10. The prompt `[AVAILABLE TOOLS]` section grows by approximately 1,200 characters (~300 tokens). At the current 50K token budget, this is 0.6% of context — negligible. The auto-generated prompt section from `ToolRegistry.generatePromptSection()` handles this without manual maintenance.

### File Structure

```
src/lib/ai/tools/capital/
├── get-fund-financials.ts         (existing)
├── get-capital-call-summary.ts    (new)
├── get-capital-call-details.ts    (new)
├── get-overdue-call-items.ts      (new)
├── get-distribution-summary.ts    (new)
└── get-distribution-details.ts    (new)
```

### Registration

Each tool is registered in `create-default-registry.ts` — one import + one `registry.register()` call per tool, as designed in R1.

### Soft Delete Awareness

- `CapitalCall` has `deletedAt` but no `notDeleted` filter in the schema relations. **Capital calls are not soft-deleted in current business logic** — they use status `CANCELLED` instead. The queries above filter by `status: { notIn: ['DRAFT', 'CANCELLED'] }` which is the correct business-level filter.
- `Distribution` has `deletedAt`. Apply `...notDeleted` filter.
- `CapitalCallItem` and `DistributionItem` have no `deletedAt` — they cascade-delete with their parent. No soft-delete filter needed.

### Test Strategy

Each tool gets a unit test following the `tool-parity.test.ts` pattern:
- Validates input schema (accepts valid input, rejects invalid)
- Validates output shape (mock Prisma, check field presence)
- Validates empty state behavior (returns summary string, not bare empty array)
- Validates currency passthrough (no hardcoded symbols in output)
